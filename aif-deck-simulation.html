02 · The problem
"Every application has an inherent amount of complexity that cannot be removed — only moved."
Larry Tesler · Computer scientist, 1980
The question is where the complexity goes. The Adaptive Interface Framework gives a specific answer.
03 · Competing demands
The same system.
Radically different people.
A CFO monitoring liquidity positions. A payment clerk processing a batch run. A treasury analyst building a forecast. An admin team setting entitlements.
Same platform. Competing demands on what it should show, how much depth it should expose, and what it should make easy.
A single fixed interface can't serve all of them well. It serves the average — which means it serves no one well.
Role A
CFO
Needs signals, not data. Exceptions, not workflows. Five minutes, not fifty.
Role B
Payment Clerk
Needs task queues, process flows, confirmation states. Depth, not summary.
Role C
Treasury Analyst
Needs data at full resolution. Workspace tools. Everything visible, reconfigurable.
04 · One scale
From monitoring
to full workspace.
Every module in the interface exists on a single scale. At one end: a status signal, one number, one decision prompt. At the other: a full workspace with data at depth and complete workflow tools.
The framework moves a module to the right point on that scale based on who is using it and what they need to do.
Minimal Maximal
Minimal
Status signal
One number. One alert. One action prompt. Nothing more.
Median
Operational view
Summary with drill-down. Task queue. Contextual actions.
Maximal
Full workspace
Data at depth. Configurable layout. Complete workflow tooling.
05 · The second scale
One dashboard
isn't enough.
The module scale tells you how deep each module goes. But a dashboard that shows everything — payments, accounts, liquidity, admin — at once collapses under its own weight, regardless of depth.
The second scale organises the interface around what users are actually doing. Four operational areas. Each with its own surface.
01
Payments
Initiation, authorisation, tracking, reconciliation.
02
Accounts
Balances, statements, position monitoring.
03
Admin
Entitlements, users, permissions, audit.
04
Liquidity
Cash positioning, forecasting, intraday flows.
06 · Both scales together
This is what makes it a
framework, not a design.
A single dashboard with personas is a design decision. Someone built it, someone maintains it, and every new user type requires a new design effort.
Two scales — module depth and operational area — mean the system infers the right interface from context. It doesn't need to be rebuilt for each new user type. It scales.
Payments
Minimal → Maximal
Accounts
Minimal → Maximal
Admin
Minimal → Maximal
Liquidity
Minimal → Maximal
Each operational area runs the full scale independently. A user in Payments at Minimal can simultaneously be in Liquidity at Maximal.

08 · How it works
Three layers
underneath.
The framework operates at three levels of the interface simultaneously. You don't need to understand all three today — but it matters that they exist, because it's what separates a framework from a set of templates.
01
Screen structure
Which surfaces exist and how they connect. The IA scales on two axes: breadth and depth.
02
Page sections
Within each screen, sections appear, collapse, or expand based on context and entitlement.
03
Components
Each component has Minimal, Median, and Maximal states. Same component, right depth.

12 · Appendix B
Frontier capabilities
  • Contextual inferenceSystem infers appropriate depth from entitlement profile, recency, and task context. No user configuration required.
  • Auto-scalingWhen a user's role changes, the interface scales without redesign. New role is a new context, not a new design brief.
  • AI-assisted depthSurfaces the right operational area and depth based on time of day, account activity, and historical patterns.
  • Cross-area persistenceContext established in one operational area informs the starting state of another.
