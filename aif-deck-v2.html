Tesler's Law  ·  1984
"Every system has an irreducible amount of complexity. The only question is who deals with it."
Larry Tesler  ·  Xerox PARC  ·  Apple Chief Scientist  ·  inventor of cut and paste
— Most interfaces answer Tesler's question the same way: the user deals with it.
— Everyone sees the same screen. The payment clerk and the treasury manager. Day one and year thirty.
— The usual fix is more screens. More versions. More to build and maintain.

AIF is a different answer. Move the complexity back where it belongs — and let the interface do the work.

02  ·  The Idea

One scale.
Every part of the interface uses it.

There is a single spectrum — from the bare minimum to the full picture. Every screen, every section, every component sits somewhere on it.


The right position isn't fixed. It's determined by who is there and what they need to do.


Not a different interface for each person. The same interface, reading the moment.

03  ·  How It Works

Three layers.
One scale.
Each layer uses it differently.

The same Minimal→Maximal scale runs through all three layers. But each layer has a different degree of freedom — how strictly it follows the scale varies. That's where the system gets both its structure and its flexibility.

04  ·  Layer 1 — Screen Structure
Entitlement-driven

The interface scales to who you are — in two directions at once.

Some users see one screen. Others see four. That's the horizontal scale — breadth. How many areas of the product a person is allowed to reach.


Within each screen, how much is shown also scales. That's the vertical scale — depth. A restricted user has one job. The screen reflects that.


"Dashboard" is not a fixed thing. For one person it's an overview of the whole business. For another it's their workspace. The framework holds both.

05  ·  Layer 2 — Page Sections
Ordered

Where a section sits tells you what it does.

Each screen is made up of sections stacked in order. That order is largely fixed — but what lives inside each section is not.


— Alert strip — at the top, always. Flags what needs attention right now. Always minimal.
— Action grid — below that. Signals and actions together. Depth varies.
— Work surface — at the bottom, optional. Full operational depth. Only appears when the role calls for it.

06  ·  Layer 3 — Components
Fluid

Same component. Three levels of depth.

Any component can run at any level. A payments widget on a monitoring screen shows just the signal. The same widget on an operations desk shows the full picture. Nothing changes except the context it's in.

07  ·  Where We Start

A tight, considered first product. Not a limited one.

The first version sits toward the left of the scale by choice. Clean structure. Four areas. Two levels of depth. Every screen built by hand.


The framework can do more than this. That's not a promise for later — it's already built in. The scope is a product decision, not a design constraint.

Appendix A  ·  Extended Context

When the interface knows who it's talking to.

Extended Context switches on when the system has entitlements data. It knows what each person can see, do, and act on. The interface shapes itself around that — without designers building separate versions for each role.


This is not a new product. It's the same framework with richer inputs.

Horizontal — who sees what
—Entitlements set how many areas a person can reach
—A restricted user has one area — that is their whole interface
—A full user sees all four areas from a true overview
Vertical — how much they see
—Components unlock to full depth for those entitled to it
—The work surface section appears only when the role warrants it
—Alerts show only what this person's role can act on
Status

Entitlements system in active development. Framework ready on both axes.

Appendix B  ·  Frontier Technologies

The interface assembles itself.

At the frontier the screen stops being a designer's decision. The system knows who you are, what you're doing, and what matters right now. It builds the right interface on the spot.


This is what Tesler's Law looks like when it's fully solved. Complexity doesn't just move to the designer — it moves to the machine. The interface carries it entirely.

What this makes possible
—Screens assembled from context, not built in advance
—The interface takes action and surfaces the result
—Silence unless something needs attention
—Depth adjusts in real time as the moment changes
Status

Speculative — but the framework is built to get here.
