The Interface Landscape

A survey of how financial services platforms handle the complexity problem — and what they reveal about the space AIF is entering.

Every platform in financial services faces the same structural challenge: the same product must serve radically different users. A CFO and a payment clerk, a senior portfolio manager and a junior analyst, a bond trader and a risk officer — they all touch the same system. Nine products across three sectors show how the market is currently answering that challenge. The pattern is consistent, and consistently incomplete.

Research was conducted to the standard set out in the brief: every claim is marked by its evidence basis. Where no public UI evidence exists, diagrams are presented as hypothesis — clearly labelled, with sources stated.

AIF comparison

Restriction vs calibration.

ACCESS is the clearest example of what the brief calls "restriction vs calibration." The entitlement system is sophisticated and well-built — but it controls access, not presentation. A restricted user gets a smaller version of the full interface rather than a simpler and more appropriate one.

AIF's claim here is specific: the widget a payment authoriser sees should look different from the one a treasury analyst sees — same data, different depth and context. ACCESS shows no evidence of this. The system trusts entitlements to do the work of design.

01

IA

Information architecture

HSBCnet follows a portal-plus-specialist-tools model. The main platform handles general access: account overview, payments navigation, reporting. Specialist functions — liquidity management, cash flow forecasting, investment solutions — exist as separate portals accessed via single sign-on. Each portal has its own depth model. The platform's coherence is single authentication, not unified surface.

02

Role

Role-awareness

Permission-based visibility. The LMD user guide states plainly: "Liquidity dashboard is only visible to users that have the appropriate permissions." Users without permissions don't see a simpler version — they don't see the LMD at all. Personalisation within the LMD is user-configured, not system-inferred.

03

Cx

Complexity handling

Segmentation. Each functional area gets its own portal with its own depth model. This keeps individual portals manageable, but multiplies the total number of surfaces the organisation must maintain — and multiplies the number of contexts a user must learn to navigate.

AIF comparison

Multi-portal proliferation.

The LMD is genuinely well-designed within its scope — real-time data, excellent drill-down from global to account, configurable view. The problem is structural: it exists separately from the platform it should be part of.

AIF's direct argument against this model is the multi-portal proliferation problem. Every new specialist function requires a new portal. Each portal diverges over time. AIF's four operational areas (Payments, Accounts, Admin, Liquidity) contain all of this in one adaptive surface — so a user moving from cash oversight to payment authorisation is navigating sections, not products.

IA

Information architecture

Aladdin is a family of applications sharing a common design system and a single data layer. A portfolio manager and a risk manager access different Aladdin applications — purpose-built for each function — not different views of the same surface. "One system, one database" refers to data architecture, not interface architecture. Role-awareness

Strongly differentiated by application. Portfolio managers, risk managers, traders, compliance teams, and operations staff each access a distinct Aladdin application tuned to their workflows. 03

Cx

Complexity handling

Specialisation. Build the right tool for each function; let the shared data layer provide coherence. The Addison AI assistant (launched late 2025) adds a context-aware natural language layer — it knows which portfolio view the user is currently in and surfaces contextually relevant answers. This is AI as a complexity mediator: rather than redesigning the interface, add an intelligent layer on top.

AIF comparison

Specialisation vs adaptation.

Aladdin operates at a different scale and in a different domain from AIF's target product. But the structural question is identical: how do you serve radically different users with the same underlying data? Aladdin's answer — specialised applications per function — delivers deep, purpose-built tools but fragments the experience. A CIO who needs cross-functional visibility is navigating between products.

The Addison AI assistant points in the same direction as AIF's frontier horizon. Context-aware assistance that already knows what you're looking at is exactly the instinct behind AIF's adaptive surface. The difference: Addison is layered on top of a fixed architecture. AIF builds context-awareness into the architecture from the start.

Admin-configured and genuinely effective — when the configuration is done well. Salesforce's platform allows different page layouts, home dashboards, and navigation items for different user profiles. AIF comparison

Administrative vs structural.

FSC's configured role-awareness is the manual equivalent of what AIF does structurally. An excellent FSC implementation achieves role-differentiated interfaces — but a Salesforce administrator built and maintains them. AIF's claim is that role-awareness should be structural, not administrative.

FSC's three-tier record page (highlights → detail → related) is structurally parallel to AIF's alert strip → action grid → work surface model, but applied at the record level rather than the platform level. AIF extends this hierarchy to the entire surface.

AIF comparison

Closest design philosophy in research.

Addepar's stated design principles are the most direct parallel to AIF's theory: "Guide beginners, empower experts — make products that are easy and intuitive but flexible enough to allow users to dive deep." That is a paraphrase of the Minimal→Maximal scale. The recognition is there. The difference is in how it's resolved.

Addepar resolves it through user configuration (you decide your depth) and progressive disclosure (the system adds depth when you navigate into it). AIF resolves it through system inference (the framework already knows your depth from context). These are different answers to the same question. Addepar's answer requires an expert user willing to configure; AIF's scales to users who don't know they need to configure anything.

AIF comparison

Same instinct. Different architecture.

Bloomberg is the most extreme example of moving complexity to the user. It works precisely because its user base is a small, expert, highly motivated cohort who use the platform continuously and can invest months in learning it. The model is categorically unsuitable for corporate banking, where users are diverse, expertise levels vary widely, and the platform must work from day one.

Bloomberg's stated design goal — "contextual relevance to the other things you're doing" — is exactly what AIF's alert strip and action grid are designed to provide structurally. Bloomberg achieves contextual relevance through user composition. AIF achieves it through system inference. Bloomberg's version requires years of training. AIF's doesn't.

Cross-category findings

What the market shows.

Nine platforms. Three sectors. The problem is the same everywhere. The answers fall into four distinct patterns — and none of them is what AIF is building.

Pattern 1

Restriction

Control what users can see through entitlements and permissions. The interface doesn't change for different users — their access does. JPMorgan ACCESS, HSBCnet, and Barclays iPortal all operate primarily in this mode. It is technically straightforward and regulatorily defensible. It does not solve the complexity problem for the user — it relocates it to the admin layer and the access control team.

Pattern 2

Designed adaptation

Build different interface states for different user types, maintain them manually. CitiDirect BE is the only platform in this research attempting genuine persona-differentiated interface design. The approach is sophisticated and, within its scope, effective. It is also brittle: every new user type requires a new design effort, and the maintenance burden compounds over time. Complexity has moved to the UX team's workload rather than to the system.

Pattern 3

User configuration

Give users tools to configure their own depth and composition. Addepar's dashboard, Salesforce FSC's Lightning layouts, LSEG Workspace's drag-and-drop modules. The right instinct — users should be able to access appropriate depth — delivered with the wrong mechanism. Users who don't know what to configure get the default. Users who misconfigure get the wrong experience. Expert users benefit; everyone else is left to figure it out.

Pattern 4

Expertise transfer

Move the complexity to the user's head. Bloomberg Terminal. This is the most extreme version, and it works for its specific user base — expert professionals who invest months in learning the platform and derive status from mastering it. It is not transferable to corporate banking, where users are diverse, expertise levels vary, and the platform must work from day one.

There is a fifth pattern beginning to emerge. Every category in this research has seen platforms add AI assistants in the last twelve to eighteen months: Aladdin Addison, Addepar Addison, Salesforce Einstein Copilot, LSEG Digest. In each case the AI is positioned as a natural language interface to a fixed platform — you can ask it questions about what you're looking at, but the platform itself doesn't change. This is the market converging on AI as a workaround for interfaces that weren't designed to be context-aware.

The AI workaround is better than nothing. It is not the same as building context-awareness into the architecture. AIF's claim is structural: the interface itself should scale to whoever is using it and whatever they need to do at that moment. Not because an AI layer interprets their questions, but because the framework already knows the answer.

The competitive landscape confirms the problem AIF is designed to solve. It also confirms that no platform in corporate banking has solved it. CitiDirect is the closest — and the gap between their approach and AIF's is the architectural argument for building AIF.

The complexity can't be destroyed. The question is who deals with it. The market's current answers: the admin team, the UX team, the user themselves, or no one. AIF's answer is the system.
